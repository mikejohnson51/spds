Jargon

As with any field, web development (or web communication!) has its own lingo and toolsets. Here is a brief summary of the unfamiliar terms you will see in this activity:

  • URL: URL stands for Uniform Resource Locator and is a reference to a web resource specifing its location on a computer network and a mechanism for retrieving it. Think of it as a web address.

  • HTML: HTML stands for Hyper Text Markup Language. HTML is the standard markup language for Web pages. Remember that we knit our markdown (Rmd) files to HTML.

  • CSS: CSS stands for Cascading Style Sheets. CSS describes how HTML elements are displayed. Should a header (#) be blue? Green? Orange?

CSS saves a lot of work because it can control the layout of multiple web pages at once.

  • YML (or YAML): YAML (stands for ‚ÄúYAML Ain‚Äôt Markup Language‚Äù) is a human-readable data-serialization language. It is commonly used for configuration files and in applications where data is being stored or transmitted. In Rmarkdown files, it describes how the file should knit.

Website Hosting

GitHub Pages

GitHub Pages is a static site hosting service that takes HTML, CSS, and JavaScript files straight from a repository and publishes them as a website. GitHub Pages is available for public repositories.

There are three types of GitHub Pages sites: project, user, and organization.

We are not interested is organization sites, so lets just focus on the details of user and project sites. In this lab, we will be building your user site, and in future labs, you might build a project site.

user sites User sites are connected to a GitHub account (not a project/repository). To publish a user site, you must create a repository named <user>.github.io. User sites (if activated) are available at http(s)://<username>.github.io.

project sites Project sites are connected to a specific project (repository) hosted on GitHub. The source files for a project site are stored in the same repository as their project. Project sites (if activated) are available at URLs following the pattern: http(s)://<username>.github.io/<repository>.

Why are Github Pages cool? Well they are free. They are secure. And like the versioned code, all you need to do to modify a sites is edit, commit, and push your changes.

Filling in our information

_site.yml

We will start by filling out the yml file. yml files basically provide a road-map telling R how to assemble a website. We use a yml file to specify how our website looks, just as we use yml in the header of an Rmarkdown file to define how our document renders. yml files are super, super sensitive to tabbing so make sure your tabbing and spacing is correct if problems arise.

We want to tell our site 4 things:

  1. What is the name of the site
  2. Where should the site HTML files knit to? (output_dir)
  3. That we want a nav-bar and
  4. What should be in the nav-bar

We can do this by filling in the appropriate yml metadata:

name: "example-website"
output_dir: "."
navbar:
  title: "Example Website"
  left:
    - text: "Home"
      href: index.html
    - text: "GIS Work"
      href: geog176A.html

In our _site.yml file, we are calling our website ‚Äúexample-website‚Äù, we will knit our files to the home (‚Äú.‚Äù) directory; and we create a navbar.

navbar references Bootstrap‚Äôs powerful, responsive navigation header. The ‚Äúnavbar‚Äù includes support for branding, navigation, and more. Bootstrap is one of the world‚Äôs most popular front-end open source toolkits and is integrated (in a limited way, with knitr/rmarkdown). We not only use Bootstrap elements here, but will apply Bootstrap themes, latter on. An example of a ‚Äúnavbar‚Äù can be seen below:

Build your site

The minimum requirement for an R Markdown website is an index.Rmd file and a _site.yml file. We have each of these! Now you can run render_site() in the console. When you do this, the following occurs:

  1. All *.Rmd and *.md files in the root directory are rendered into HTML.
    • Markdown files beginning with **_** are not rendered (a convention to designate files that are to be included by top level Rmd documents as child documents).
  2. The generated HTML files and any supporting files (such as CSS and JavaScript) are copied into an output directory (_site/ by default).
    • However! If you look at the yml file we created, we OVERRODE this default by defining the output_dir as ‚Äú.‚Äù -or- the root project directory.

##  [8] "united"    "cosmo"     "lumen"     "paper"     "sandstone" "simplex"   "yeti"

To control the output styles of an HTML document in R Markdown, we must specify the output format (html_document) in the YAML metadata. Only by doing this can we specify the theme we want to use. For an example see our modified _site.yml below, again remembering that the tabbing is sensitive:

name: "example-website"
output_dir: "."
navbar:
  title: "Example Website"
  left:
    - text: "Home"
      href: index.html
    - text: "GIS Work"
      href: geog176A.html
output:
  html_document:
    theme: darkly

Test a few different themes to find the one you like!
